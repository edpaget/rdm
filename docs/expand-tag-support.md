# Tags on roadmaps, phases, and tasks

## Overview

Tags are short, user-defined labels you can attach to any roadmap, phase, or task. They give you a flat, lightweight way to mark related work — `bug`, `auth`, `tech-debt`, `ui` — independent of which roadmap or phase it lives in.

Before this work, only tasks could carry tags. Now roadmaps and phases support the same `tags` field, every list/show/search surface honors tag filters, and the `rdm` CLI, HTTP server, and MCP server expose consistent ways to set, replace, clear, and filter by tags. Agent-facing instructions generated by `rdm agent-config` teach LLMs to use tags too.

## Motivation

A roadmap-and-phase model groups work hierarchically, which is good for execution order but bad for cross-cutting concerns. If you have three roadmaps that each include some auth-related work, there's no easy way to ask "what auth work is in flight?" Tags make that question one command away. They also let downstream surfaces (the HTML server, search, MCP) filter by the same vocabulary the team already uses in commit messages and PR titles.

## Usage

### Creating items with tags

`--tags` is comma-separated on every create command. Pass it on `update` to replace the existing list, or `--tags ""` to clear.

```bash
rdm roadmap create auth-overhaul --title "Auth overhaul" --tags auth,security --no-edit --project myproj
rdm phase create new-jwt --title "Switch to JWT" --number 1 --tags auth --no-edit --roadmap auth-overhaul --project myproj
rdm task create rotate-keys --title "Rotate signing keys" --tags auth,ops --no-edit --project myproj

# Replace existing tags
rdm task update rotate-keys --tags auth --no-edit --project myproj
# Clear tags
rdm task update rotate-keys --tags "" --no-edit --project myproj
```

Tags appear in `roadmap show` / `phase show` / `task show` (text, JSON, and Markdown formats). They round-trip through markdown frontmatter as `tags: [a, b]` and persist across reads/writes.

### Listing by tag

The `task list` command's existing `--tag` filter has been mirrored on roadmap and phase listings via the HTTP and MCP layers, plus a uniform `--tag` on `rdm search` (see below).

```bash
# Tasks with a specific tag (existing behavior, unchanged)
rdm task list --tag bug --project myproj
```

### Searching with tag filters

`rdm search` accepts repeatable `--tag` flags that AND together. An item with no tags is excluded from any non-empty filter.

```bash
# Find every roadmap, phase, or task tagged "bug"
rdm search "" --tag bug --project myproj

# Combine with the fuzzy text query
rdm search auth --tag bug --project myproj

# Multiple tags AND together — must carry every listed tag
rdm search auth --tag bug --tag ui --project myproj

# Combine with --type / --status / --format as before
rdm search "" --tag bug --type task --format json --project myproj
```

The JSON output includes a `tags` array on each result (omitted when the item has no tags).

### HTTP server

`rdm serve` exposes `?tag=` on the roadmap, phase, and (existing) task list endpoints, plus a new dedicated phase-list endpoint:

```
GET /projects/<p>/roadmaps?tag=auth                          → roadmaps with tag "auth"
GET /projects/<p>/roadmaps/<r>?tag=audit                     → roadmap detail; embedded phases section filtered to "audit"
GET /projects/<p>/roadmaps/<r>/phases                        → list phases (HAL JSON)
GET /projects/<p>/roadmaps/<r>/phases?tag=audit              → phases tagged "audit"
GET /projects/<p>/tasks?tag=bug                              → tasks tagged "bug" (unchanged)
```

JSON responses for roadmaps and phases now include a `tags` array on summaries and details.

`POST` and `PATCH` request bodies for roadmaps and phases accept a `tags: [...]` array. On `PATCH`, set `clear_tags: true` to remove all tags; passing both `tags` and `clear_tags: true` returns `422 Unprocessable Entity`.

```http
POST /projects/myproj/roadmaps
Content-Type: application/json

{"slug": "auth-overhaul", "title": "Auth overhaul", "tags": ["auth", "security"]}
```

```http
PATCH /projects/myproj/roadmaps/auth-overhaul
Content-Type: application/json

{"clear_tags": true}
```

#### Quick-filter chips (HTML views)

The HTML roadmap, phase, and task list views render a chip group above the table for one-click filtering by tag. Define the chips per plan-repo in `rdm.toml`:

```toml
[[server.quick_filters]]
label = "Bugs"
tag = "bug"

[[server.quick_filters]]
label = "UI work"
tag = "ui"
```

Override at run time with the env var or repeatable CLI flag — higher levels fully replace lower ones (no merge):

```bash
RDM_SERVER_QUICK_FILTERS="Bugs:bug,UI work:ui" rdm serve
rdm serve --quick-filter Bugs:bug --quick-filter "UI work:ui"
```

Precedence: `--quick-filter` flags > `RDM_SERVER_QUICK_FILTERS` env var > `[server.quick_filters]` in `rdm.toml`. Each chip links to the same page with `?tag=<value>`; the active chip is visually highlighted, and an "All" link appears next to the chips when a filter is active so users can clear it in one click.

### MCP server

Every MCP tool that touches tags is updated:

| Tool | New parameter |
| --- | --- |
| `rdm_roadmap_create` | `tags: ["bug", "ui"]` |
| `rdm_roadmap_update` | `tags: [...]` (replace), `clear_tags: true` (remove all) |
| `rdm_roadmap_list` | `tag: "bug"` (filter) |
| `rdm_phase_create` | `tags: ["audit"]` |
| `rdm_phase_update` | `tags: [...]`, `clear_tags: true` |
| `rdm_phase_list` | `tag: "audit"` |
| `rdm_search` | `tags: ["bug", "ui"]` (AND across tags) |
| `rdm_task_create` | `tags: [...]` *(existing — example added to schema description)* |
| `rdm_task_update` | `tags: [...]` *(existing — pass `[]` to clear)* |

Schema descriptions on each new field tell MCP clients the semantics (replace vs. clear, AND across multi-tag filters, exclusion of items with no tags).

### Agent-facing instructions

`rdm agent-config` (both CLI and `--mcp` modes) now writes instruction templates that demonstrate every tag-related command, plus a short tagging-convention block:

- Lowercase kebab-case (`bug`, `auth`, `tech-debt`).
- Prefer existing tags — check with `rdm search "" --tag <candidate>` (or the MCP equivalent) before inventing a new one.
- Tag work to make it findable across roadmaps, phases, and tasks.

The in-repo Claude Code skills (`.claude/skills/rdm-tasks`, `.claude/skills/rdm-roadmap`) inherit the same examples, so agents driving the rdm source repo also tag their work.

## How it works

- **Data model**: `Phase` and `Roadmap` in `rdm-core` gained a `tags: Option<Vec<String>>` field with the same `serde(skip_serializing_if = "Option::is_none")` pattern already used on `Task`. Frontmatter round-trips unchanged for items with no tags. Existing tag-less files still parse, so this is a backward-compatible addition.
- **Promote**: `rdm promote <task>` carries the source task's tags onto the seed phase of the new roadmap, so context isn't lost on conversion.
- **Search**: `SearchFilter.tags: Option<Vec<String>>` is a hard pre-filter applied before fuzzy scoring. AND semantics — every listed tag must be present. `Some(vec![])` and `None` both mean "no tag constraint".
- **Server config**: a new `[server]` table in `rdm.toml` with a `quick_filters` list. Resolved per-request via the new `AppState.quick_filters` field. Full replacement (not merge) at every precedence level so you can override the project default for a one-off run.
- **Clear semantics**: across all surfaces, the core ops treat `Some(empty Vec)` as "clear" and `None` as "preserve". The HTTP and MCP layers expose this as a `clear_tags: true` boolean (because JSON omits-vs-empty is awkward), and the CLI exposes it as `--tags ""`.

## Limitations

- **Flat namespace**: tags are unstructured strings. There's no hierarchy, no aliasing, and no rename-with-history. If you change a convention (`tech-debt` → `techdebt`), you'll need to update affected items individually.
- **No autocomplete or registry**: nothing prevents typos or near-duplicates (`auth` vs. `authn`). The convention block in agent instructions and the `search "" --tag <candidate>` check are conventions, not enforcement.
- **No migration tooling**: this work adds the field; it doesn't backfill tags onto pre-existing items. You opt in by editing each item or via `update --tags`.
- **HTML phase-list page**: the new `GET /roadmaps/<r>/phases?tag=` endpoint returns HAL JSON. Browsers requesting `text/html` are redirected to the roadmap detail page (which renders the embedded phases table with the same `?tag=` filter) — there is no standalone phase-list HTML view.
- **Search performance**: tag matching is currently O(items × required-tags) and clones each surviving item's tag list once. For very large plan repos this is followed up by `task/search-defer-tag-clone`, which defers the clone until after the relevance cutoff trims results.
